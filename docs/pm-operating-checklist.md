# PM 운영 체크리스트

PM 레인은 기능을 많이 만드는 역할이 아니라, 대학생에게 실제로 쓸 이유가 있는 MVP를 좁게 유지하고 검증하는 역할이다.

## PM의 핵심 책임

- MVP 문제 정의 유지
- 사용자 대상과 사용 상황 명확화
- 기능 우선순위 결정
- 작업별 수용 기준 작성
- 데이터 품질 기준 관리
- 안전 수집 원칙 준수 확인
- 베타 사용자 피드백 수집
- 지표 기반 Go/No-Go 판단
- 범위 확장 요청 관리

## 현재 제품 정의

대학생이 직무 키워드와 마감일 기준으로 신뢰할 수 있는 채용 공고를 찾고, 원문을 확인하고, 관심 공고를 저장할 수 있게 한다.

MVP는 채용 플랫폼 전체가 아니라 공고 탐색/저장 도구다.

## PM 진행사항

### 2026-06-26 기준

- MVP 범위는 `대학생용 신뢰 가능한 공고 탐색/저장 도구`로 유지한다.
- 사용자 검색 시점의 외부 사이트 실시간 크롤링은 제외하고, 주기 수집 후 내부 DB를 검색하는 방식으로 확정했다.
- 무료 운영 우선 구조를 확정했다: Cloudflare Pages, Render Free, Supabase Free, GitHub Actions.
- 프론트엔드는 React + Vite + TypeScript로 전환했고 Cloudflare Pages 루트 빌드가 `dist/`를 생성하도록 정리했다.
- Cloudflare Pages 배포 전용 설정은 `Build command: npm run build`, `Build output directory: dist`로 정리했다.
- API 미배포 또는 API 연결 실패 상태에서도 화면 검증이 가능하도록 Web 내장 샘플 공고 fallback을 추가했다.
- FastAPI는 `GET /health`, `GET /jobs`, `GET /jobs/{id}`를 제공하고, `sample`/`database` 데이터 소스 전환을 지원한다.
- PostgreSQL `jobs` 테이블 migration, sample seed, DB-backed 검색 경로를 추가했다.
- Render API 배포용 `render.yaml`과 Supabase/Render/Cloudflare/GitHub Actions 연결 절차 문서를 추가했다.
- GitHub Actions workflow를 추가했다: `Bootstrap database`, `Collect jobs`, `API smoke test`.
- Crawler는 사용자 검색과 분리된 독립 CLI로 준비했고, 초기 수집 대상은 Greenhouse/Lever 공개 채용 API allowlist로 제한했다.
- 현재 외부 콘솔 작업이 남아 있다: Supabase DB 생성, GitHub `DATABASE_URL` secret 등록, `Bootstrap database` 실행, Render API 배포, Cloudflare `VITE_API_BASE_URL` 설정.
- 현재 주요 리스크는 실제 DB/API runtime 검증 미완료, 첫 실제 수집 소스 미확정, 무료 티어 sleep/한도, Python 의존성 설치 환경의 DNS 문제다.

## P0 수용 기준

### 공고 목록

- 사용자는 공고 제목, 회사명, 직무 키워드, 마감일, 출처, 수집 시각을 볼 수 있다.
- 사용자는 직무 키워드로 공고를 검색할 수 있다.
- 사용자는 마감일 기준으로 공고를 필터링할 수 있다.
- 사용자는 원문 링크로 이동할 수 있다.

### 공고 상세

- 사용자는 공고의 핵심 요약 정보를 볼 수 있다.
- 사용자는 원문 URL과 원문 확인 시각을 볼 수 있다.
- 공고 본문 전문, 이미지, 첨부파일은 저장하거나 재게시하지 않는다.

### 관심 공고

- 사용자는 관심 공고를 저장하고 해제할 수 있다.
- 초기 구현은 로그인 없이 브라우저 localStorage로 처리한다.
- 저장 기능 사용률을 베타 검증 지표로 기록한다.

### 수집

- 사용자의 검색 버튼은 외부 크롤링을 실행하지 않는다.
- Source Registry에서 허용된 소스만 수집한다.
- 수집 시각, 출처, 원문 URL을 저장한다.

## 기능 시작 전 PM 체크

새 기능을 시작하기 전에 아래 질문에 답한다.

- 관련 요구사항 ID는 무엇인가?
- 이 기능이 P0/P1/P2 중 어디에 속하는가?
- 대학생 사용자가 이 기능을 왜 써야 하는가?
- 기능 완료를 어떻게 판단할 것인가?
- 어떤 검증 기준으로 완료를 확인할 것인가?
- 어떤 지표에 영향을 줄 것인가?
- 데이터 수집/법적/운영 리스크가 있는가?
- 이번 작업에서 제외할 범위는 무엇인가?

## 주간 PM 체크포인트

| 주기 | 확인 항목 |
|---|---|
| 매 작업 완료 | `PROJECT_STATUS.md`에 변경, 검증, 다음 작업 기록 |
| 매주 | MVP 지표와 리스크 점검 |
| 크롤링 소스 추가 전 | `docs/crawling-policy.md` 기준 검토 |
| 배포 전 | 무료 한도, 응답 지연, secret 관리 확인 |
| 베타 전 | 테스트 사용자 시나리오와 피드백 질문 준비 |

## 베타 검증 질문

- 검색/필터가 설명 없이 이해되는가?
- 공고 상세에서 원문으로 이동할 이유가 생기는가?
- 관심 저장이 다시 방문할 이유를 만드는가?
- 공고가 최신이라고 믿을 수 있는가?
- 출처와 수집 시각 표시가 충분한가?
- 부족한 직무/소스/필터는 무엇인가?

## 범위 확장 방지 원칙

아래 기능은 사용자 행동 지표가 확인되기 전까지 보류한다.

- AI 추천
- 자기소개서/면접 기능
- 커뮤니티
- 학교/전공별 개인화
- 이메일/카카오 알림
- 로그인 기반 동기화
- 대형 민간 플랫폼 무단 크롤링

PM은 각 개발 작업이 MVP 검증에 직접 기여하는지 계속 확인한다.
