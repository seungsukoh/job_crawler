# 엔지니어링 원칙

이 프로젝트는 빠르게 만들되, 이후 유지보수가 어려워지는 구조를 피한다.

## 핵심 원칙

- 작게 만든다.
- 경계를 분리한다.
- 계약을 문서화한다.
- 안전한 기본값을 둔다.
- 배포 플랫폼에 과하게 묶이지 않는다.
- 크롤링은 보수적으로 운영한다.
- 테스트 가능한 단위로 나눈다.

## 구조 원칙

### Web

- `apps/web`은 화면과 사용자 상호작용을 담당한다.
- API 주소는 환경 변수로만 받는다.
- API 응답 형태는 별도 client/type 계층에서 다룬다.
- 초기 관심 공고 저장은 localStorage로 제한한다.

### API

- `apps/api`는 조회 API와 서버 설정을 담당한다.
- route는 리소스별 파일로 나눈다.
- DB 접근 로직은 route 함수 안에 직접 섞지 않는다.
- schema, repository, service 계층은 필요해질 때 추가한다.
- `/health`처럼 운영에 필요한 endpoint는 간단하고 안정적으로 유지한다.

### Crawler

- `crawler`는 API 서버와 분리된 CLI로 유지한다.
- Source Registry를 통과하지 못한 소스는 실행하지 않는다.
- 수집 결과는 정규화된 형태로 저장한다.
- 외부 사이트 변경이나 차단이 API 서버 안정성에 영향을 주지 않게 한다.

### Infra

- `infra`는 로컬 실행과 배포 보조 설정을 둔다.
- 환경 변수 이름은 `.env.example`에 먼저 반영한다.
- secret 값은 코드와 문서에 기록하지 않는다.

## 유지보수 기준

- 기능 하나는 가능한 한 한 레인 안에서 끝낸다.
- 여러 레인을 건드릴 때는 API contract, env var, DB schema 같은 공유 계약을 먼저 갱신한다.
- 새 의존성은 필요한 이유가 분명할 때만 추가한다.
- 공통화는 반복이 실제로 생긴 뒤에 한다.
- 대규모 리팩터링은 기능 작업과 분리한다.
- 문서 변경 없이 중요한 정책이나 구조를 바꾸지 않는다.

## 테스트 기준

- API endpoint는 최소 happy path test를 둔다.
- DB migration이 생기면 migration 적용/롤백 가능성을 확인한다.
- 크롤러는 외부 요청을 직접 때리는 테스트보다 parser/normalizer 단위 테스트를 우선한다.
- 프론트엔드 주요 화면은 구현 후 브라우저로 확인한다.

## 확장 기준

아래 조건이 생기기 전까지 복잡한 구조를 도입하지 않는다.

- 인증: 관심 저장이나 알림이 반복 사용 지표로 검증된 뒤
- 별도 검색 엔진: PostgreSQL 검색으로 부족한 실제 지표가 확인된 뒤
- 상시 worker: GitHub Actions 수집으로 부족한 실행 빈도와 안정성 문제가 확인된 뒤
- 유료 인프라: 무료 한도가 사용자 경험이나 수집 안정성을 실제로 막은 뒤

## 금지할 방향

- 사용자 검색 시점 외부 크롤링
- route 함수 안에 수집/파싱/DB/응답 로직을 모두 넣는 구조
- 배포 플랫폼 전용 기능에 핵심 로직을 묶는 구조
- 테스트 없이 DB schema를 바꾸는 작업
- Source Registry 없이 새 크롤러를 붙이는 작업
- MVP 지표와 무관한 대형 기능 추가
