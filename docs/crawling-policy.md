# 안전 수집 정책

## 핵심 원칙

채용 공고 수집은 `명시적으로 허용된 경로 우선, 불명확하면 수집하지 않음`을 기준으로 한다. MVP 단계에서는 데이터 양보다 법적/운영 리스크를 낮추는 것이 더 중요하다.

사용자 검색 요청은 외부 사이트 크롤링을 유발하지 않는다. 크롤링은 사전에 정한 스케줄, GitHub Actions, 관리자 수동 실행으로만 수행하고, 사용자는 이미 저장되고 검증된 내부 DB를 검색한다.

## 허용 소스 우선순위

1. 공식 API
2. 공식 RSS/Atom 피드
3. 직접 등록 공고
4. robots.txt와 약관 검토 후 허용된 공개 HTML 페이지

## MVP에서 제외하는 소스

- 민간 대형 채용 플랫폼의 공고 상세 페이지 무단 크롤링
- 로그인 후 접근 가능한 정보
- 유료 회원 전용, 제휴사 전용, 검색 결과 제한 우회
- CAPTCHA, 봇 차단, 동적 토큰, 앱 API 역공학이 필요한 소스
- 약관상 스크래핑, 복제, 재배포, 데이터베이스화가 금지된 소스
- 개인정보가 과도하게 포함된 공고
- 사용자 검색어를 기반으로 실시간 수집해야만 결과를 만들 수 있는 소스

## 소스 등록 체크리스트

소스를 추가하기 전에 아래 항목을 기록한다.

- 소스명
- 운영 주체
- URL
- 수집 대상 경로
- 수집 방식: `api`, `rss`, `html`, `direct`
- 검토일
- 검토자
- 재검토 예정일
- robots.txt 존재 여부
- 대상 User-agent 규칙
- 수집 대상 경로의 Allow/Disallow
- Crawl-delay 존재 여부
- 약관 URL
- 자동화 접근 금지 조항 여부
- 데이터 복제/저장/재배포 금지 여부
- 상업적 이용 금지 여부
- 출처 표시 의무
- 삭제 요청 또는 권리 침해 신고 절차
- 로그인/CAPTCHA/봇 차단 여부

## 소스 판정

- `allowed`: API/RSS 또는 명시적으로 허용된 공개 경로
- `conditional`: robots 허용, 약관 금지 없음, 낮은 빈도 수집 가능
- `paused`: 약관 해석 불명확, 운영자 확인 필요
- `blocked`: 약관/robots/기술적 차단 중 하나라도 명확히 반대

`paused`, `blocked` 소스는 수집하지 않는다.

## 방식별 정책

### API

허용:

- 공식 API 키 발급 후 문서화된 엔드포인트 사용
- 호출 한도 준수
- 필요한 채용 정보만 저장
- 출처 URL과 API 제공자 표시

금지:

- 비공개 API 추정 호출
- 웹/앱 트래픽 분석으로 얻은 내부 API 사용
- 호출 한도 우회

### RSS/Atom

허용:

- 피드 주기보다 과도하지 않은 간격으로 갱신
- 제목, 요약, 날짜, 원문 링크 저장
- 출처 표시

금지:

- 피드에 없는 상세 내용을 별도 대량 크롤링
- 원문 링크 제거
- 출처를 숨기는 재게시

### 공개 HTML

허용:

- robots 허용 및 약관 금지 없음
- 낮은 빈도 수집
- 제목, 회사명, 지역, 고용형태, 마감일, 원문 URL 등 최소 필드 저장
- 소스별 동시성 1부터 시작

금지:

- 검색 결과 페이지 대량 페이지네이션
- 로그인, 쿠키 우회, CAPTCHA 회피
- 프록시/UA 회전을 통한 차단 우회
- 원문 HTML/이미지/첨부파일 대량 저장
- 담당자 개인 연락처 수집 확대

### 직접 등록

허용:

- 등록자 인증
- 게시 권한 확인
- 삭제/수정 기능 제공
- 공고 만료일 필수
- 개인정보 최소 수집

금지:

- 제3자 플랫폼 공고를 복사해서 직접 등록
- 담당자 개인 연락처 무단 공개
- 만료 공고 장기 노출

## 수집 파이프라인 원칙

1. Source Registry에서 소스 상태를 확인한다.
2. `allowed`, `conditional` 소스만 수집한다.
3. robots.txt와 약관 검토 만료일이 지난 소스는 수집하지 않는다.
4. 도메인별 rate limit을 적용한다.
5. User-Agent에는 서비스명과 연락처를 명시한다.
6. 실패 시 무한 재시도하지 않는다.
7. 원문 URL, 수집 시각, 출처명을 저장한다.
8. 공고 본문 전문, 이미지, 첨부파일은 저장하지 않는다.
9. 만료 공고는 기본 비노출한다.
10. 삭제 요청이 들어오면 보수적으로 처리한다.

## 검색과 수집 분리

허용:

- GitHub Actions scheduled workflow로 하루 1~4회 수집
- 관리자 수동 `crawl now` 실행
- 새 소스 등록 후 검증용 단발 수집
- 마감 3일 이내 공고의 별도 재검증
- 사용자가 검색하면 내부 DB의 `job_postings`를 조회

금지:

- 사용자가 검색 버튼을 누르는 시점에 외부 사이트 호출
- 검색어마다 채용 플랫폼 검색 결과 페이지 크롤링
- 인기 검색어 기반 자동 외부 크롤링
- 여러 사용자 검색 요청에 따른 병렬 크롤링
- 검색 결과가 없다는 이유로 즉시 외부 수집 실행

권장 기본 스케줄:

```text
09:00 신규 공고 수집
13:00 주요 소스 업데이트
18:00 오후 업데이트
23:00 마감/삭제 재확인
```

무료 운영 단계에서는 소스 수와 실행 시간을 고려해 하루 1~2회부터 시작하고, 필요할 때만 4회까지 늘린다.

## 자동 중단 기준

아래 조건 중 하나라도 발생하면 해당 소스 수집을 중단한다.

- robots.txt가 대상 경로를 Disallow로 변경
- HTTP 401/403/429 반복 발생
- CAPTCHA 또는 봇 차단 페이지 감지
- 요청 실패율 임계치 초과
- 예상보다 많은 페이지를 따라가는 크롤링 루프 감지
- 약관 검토 만료일 초과
- 소스 상태가 `paused` 또는 `blocked`로 변경

## 삭제 요청 프로세스

1. 삭제 요청 전용 이메일 또는 폼으로 접수한다.
2. 명백한 권리자 요청이거나 개인정보 이슈면 즉시 비공개 처리한다.
3. 원문 존재 여부, 수집 경로, 저장 데이터를 확인한다.
4. 개별 공고 삭제 또는 비공개 처리한다.
5. 필요 시 동일 소스 수집을 일시 중단한다.
6. 재수집 방지를 위해 blocklist에 등록한다.
7. 처리 결과를 요청자에게 회신한다.
8. 요청일, 요청자, 대상 공고, 처리 결과, 처리자를 기록한다.

목표 처리 시간:

- 접수 확인: 1영업일 이내
- 처리 결과: 3영업일 이내

## 개발 완료 기준

크롤링 관련 기능은 아래 조건을 만족해야 완료로 본다.

- 대상 소스가 allowlist에 등록되어 있다.
- robots.txt 확인 결과가 기록된다.
- 요청 간격과 동시성 제한이 있다.
- 실패 시 무한 재시도하지 않는다.
- 수집 데이터의 출처 URL과 수집 시각이 저장된다.
- 중복 저장 방지 로직이 있다.
- 로그에 민감 정보가 남지 않는다.
