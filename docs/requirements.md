# 요구사항 정리

이 문서는 PM 관점의 요구사항 기준점이다. 검증 기준은 이 문서의 요구사항 ID를 기준으로 작성한다.

## 제품 목표

대학생이 직무 키워드와 마감일 기준으로 신뢰할 수 있는 채용 공고를 찾고, 원문을 확인하고, 관심 공고를 저장할 수 있게 한다.

## 사용자

- 취업 준비를 시작한 대학생
- 인턴/신입/채용연계형 공고를 찾는 대학생
- 여러 사이트를 직접 돌며 마감일을 확인하는 시간이 부담스러운 사용자

## 기능 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-001 | 사용자는 공고 목록을 볼 수 있다. | P0 |
| FR-002 | 공고 목록에는 제목, 회사명, 직무 키워드, 지역, 고용형태, 마감일, 출처, 수집 시각이 표시된다. | P0 |
| FR-003 | 사용자는 직무 키워드로 공고를 검색할 수 있다. | P0 |
| FR-004 | 사용자는 마감일 또는 기간 기준으로 공고를 필터링할 수 있다. | P0 |
| FR-005 | 사용자는 공고 상세를 볼 수 있다. | P0 |
| FR-006 | 사용자는 원문 링크로 이동할 수 있다. | P0 |
| FR-007 | 사용자는 관심 공고를 저장하고 해제할 수 있다. | P0 |
| FR-008 | 관심 공고 저장은 MVP에서 로그인 없이 브라우저 localStorage로 동작한다. | P0 |
| FR-009 | 사용자는 마감 임박 공고를 구분할 수 있다. | P0 |
| FR-010 | 사용자는 공고의 수집 시각과 원문 확인 시각을 볼 수 있다. | P0 |
| FR-011 | 사용자 검색은 외부 사이트 크롤링을 실행하지 않고 내부 DB만 조회한다. | P0 |
| FR-012 | 크롤링은 스케줄러 또는 관리자 수동 실행으로만 수행한다. | P0 |
| FR-013 | 수집 소스는 Source Registry에 등록되고 허용 상태인 경우에만 실행된다. | P0 |
| FR-014 | 관리자는 수집 실행 결과와 실패 여부를 확인할 수 있다. | P1 |
| FR-015 | 사용자는 저장 공고 상태를 `검토 중`, `지원 예정`, `지원 완료`로 관리할 수 있다. | P1 |

## 데이터 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| DR-001 | 공고에는 원문 URL, 출처명, 수집 시각, 원문 확인 시각이 저장된다. | P0 |
| DR-002 | 중복 공고는 사용자 노출 기준으로 병합되거나 중복 표시되지 않는다. | P0 |
| DR-003 | 공고 본문 전문, 이미지, 첨부파일은 저장하지 않는다. | P0 |
| DR-004 | 마감된 공고는 기본 목록에서 제외하거나 명확히 구분한다. | P0 |
| DR-005 | 삭제 요청 또는 중단 대상 공고는 재수집되지 않게 처리한다. | P1 |

## 시스템 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| SR-001 | API는 `GET /health`로 상태를 확인할 수 있다. | P0 |
| SR-002 | 프론트엔드는 API base URL을 환경 변수로 설정한다. | P0 |
| SR-003 | 백엔드는 표준 FastAPI ASGI 앱으로 실행된다. | P0 |
| SR-004 | DB는 표준 PostgreSQL을 기준으로 설계한다. | P0 |
| SR-005 | DB schema 변경은 migration으로 관리한다. | P1 |
| SR-006 | 크롤러는 API 서버와 분리된 독립 CLI로 실행된다. | P0 |
| SR-007 | secret 값은 코드와 문서에 실제 값으로 기록하지 않는다. | P0 |

| ID | 요구사항 | 우선순위 |
|---|---|---|

| NFR-005 | 사용자가 보는 공고 데이터의 신뢰도를 위해 출처와 수집 시각을 표시한다. | P0 |
| NFR-006 | 주요 작업 완료 시 `PROJECT_STATUS.md`와 필요 시 `SESSION_HANDOFF.md`를 갱신한다. | P0 |
